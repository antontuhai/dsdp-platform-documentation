1. Overview

You can deploy a demo registry on the Platform that includes reference examples for modeling registry regulations. These regulations follow the same structure as the standard regulations used in any registry deployed on the Platform.

The demo registry includes:

  • Reference examples: prefixed with reference-

  • Testing examples: prefixed with feature-

These can include:

  • .bpmn business process diagrams

  • .json forms for submitting process data

  • .xml schemas for deploying the registry data model

  • and more

These reference examples and other provided samples are developed by the Platform core team. They are regularly updated and expanded with each new release, helping you:

  • Stay up to date with the latest practices and trends.

  • Experiment and test various scenarios in a controlled environment.

2. Deploying the demo registry and regulations

Follow these steps to deploy the demo registry and copy the sample regulations.

Step 1: Create the demo registry

Create a new registry named demo by following the instructions at: Deploying a registry.

To complete this process, make sure the Platform administrator is added as a registry administrator during creation under the Administrators section.

When deploying in the global region, sections like Key data and Data for signature verification do not include region-specific keys or certificates.

  1. In Gerrit, go to Browse  Repositories and use the Filter field to find the global-registry-demo-regulation repository.

Step 6: Copy the demo regulations content

On your local machine, copy the content of the global-registry-demo-regulation repository and paste it (replacing contents) into the registry-regulations repository.

Make sure to copy only the content, without the .git system folder.

Step 7: Publish the changes

At this step, you are populating the registry-regulations online Gerrit repository.

  1. Publish the changes to the demo registry regulations. After publishing, the regulation entitiesâ€”such as the data model, business processes, and formsâ€”will be available for use in the user portals:

    • Administrative portal (admin-portal)

    • Officer portal (officer-portal)

    • Citizen portal (citizen-portal)

  2. Prepare a commit and push it to the registry-regulations repository. In Git Bash, run the following commands one by one:

    Add changes to the Stage
    git add --all

    This command stages all new, modified, or deleted files in the current directory and its subdirectories.

    Commit changes
    git commit -m "added demo registry data"

    This command creates a new commit with the staged changes. The -m flag lets you add a short commit message describing the changes (in this case: "added demo registry data").

    Push changes
    git push origin HEAD:refs/for/master

    This command pushes the changes to the remote repository. Here, origin refers to the remote repository, and HEAD:refs/for/master pushes the current branch for code review before merging into master. This is the Gerrit-specific way of submitting changes for review.

    Figure 12. Git Bash. Add, commit and push

Step 8: Complete the Gerrit review and merge

  1. After pushing the changes, follow the Gerrit link shown in the terminal.

    The registry-specific Gerrit URL will look like:

    https://admin-tools-<openshift-project-name>.<dns-wildcard>/gerrit

    • <openshift-project-name> â€” your registryâ€™s name (in this example, demo)

    • <dns-wildcard> â€” the OpenShift environmentâ€™s domain name

    • gerrit â€” the endpoint leading to the Gerrit service in your registry.

  2. Wait for the system code review pipeline MASTER-Code-review-registry-regulations to complete. You can check progress at the bottom of the Gerrit page. After a successful check, your change request will receive the Verified +1 status.

  3. As a moderator, approve the change by clicking the Code-Review+2 button.

    Figure 13. Gerrit. Code review process

  4. Apply the changes to the master branch by clicking Submit (this performs a git merge).

    This triggers the automatic publishing pipeline â€” MASTER-Build-registry-regulations. You can monitor the deployment progress in Gerrit.

    Once published successfully, the demo registry will include reference examples (prefixed with reference-) and testing examples (prefixed with feature-).

To allow an officer to access the relevant reference processes in the Officer portal, you need to:

3.2. Reviewing roles and permissions

Roles are defined in the roles/*.yml files:

  • Officer roles are listed in roles/officer.yml.

  • Citizen roles are listed in roles/citizen.yml.

Example 1. Demo registry officer roles
roles/officer.yml
roles:
  - name: officer
    description: Officer role

Authorization rules are set in the bp-auth/*.yml files:

  • Officer access is defined in bp-auth/officer.yml.

  • Citizen access is defined in bp-auth/citizen.yml.

  • External system access is defined in bp-auth/external-system.yml.

Example 2. Authorization. Process-to-role mapping example
bp-auth/officer.yml
authorization:
  realm: "officer"
  process_definitions:
    - process_definition_id: 'reference-officer-view-driving-licence'
      process_name: 'View digital driving licence'
      process_description: 'View digital driving licence'
      roles:
        - officer
    - process_definition_id: 'reference-officer-driving-licence-excerpt'
      process_name: 'Excerpt digital driving licence'
      process_description: 'Excerpt digital driving licence'
      roles:
        - officer
